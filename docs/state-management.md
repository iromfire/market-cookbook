# Управление состоянием <Badge type="warning" text="Не подтверждено командой" />

## Когда использовать стор

✅ Создавай стор, когда состояние используется во многих компонентах, содержит сложную бизнес-логику и требует централизованного управления обновлением

::: info ℹ️

Например, таб «Договор» в закупке

:::

❌ Не создавай стор без надобности

::: info ℹ️

Например, на странице «Отслеживание заявки», где данные просто запрашиваются с BE и отображаются, их можно передать в дочерние компоненты через input, если это не приводит к props drilling

:::

## Выбор инструмента

✅ Используй `@ngrx/signals` [Документация](https://ngrx.io/guide/signals/signal-store)

❌ Не используй самописные решения

## Структура стора

✅ Все данные фичи храни в одном сторе

❌ Не создавай сторы, которые хранят одно поле

## Источник истины

✅ Делай единый источник истины

❌ Не создавай несколько источников истины

::: info ℹ️

Например, одни и те же данные одновременно хранятся и в сторе, и в form-сервисе

:::

## Границы ответственности

✅ Храни в сторе только состояние, а не UI-детали

❌ Не клади в стор временные флаги UI (hover, opened, selected и т.д.)

## API стора

✅ Держи API стора простым и понятным

❌ Не раздувай стор десятками методов без необходимости
